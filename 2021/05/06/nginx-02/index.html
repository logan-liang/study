Nginx核心模块

2021-05-06 Nginx Nginx Comments

主模块

包括一些nginx的基本控制功能

指令

  • daemon：daemon on |off

生产环境中不要使用daemon和master_process指令，这些指令仅用于开发调试。虽然可以使用daemon off在生产环境中，但对性能提升没有任何帮助。

  • env：env VAR|VAR=VALUE

这个命令允许其限定一些环境变量的值。默认值：TZ

  • debug_points：debug_points stop

在nginx内部有很多断言，如果debug_points的值设为stop时，那么触发断言时将停止Nginx并附加调试器。如果debug_points的值为abort，那么触发断言时将创建内核文件。默认值：none（无）

  • error_log：error_log file [ debug | info | notice | warn | error | crit ]

指定Nginx服务（与FastCGI）错误日志文件位置。可以设置等级。默认值：${prefix}/logs/error.log

  • log_not_found：log_not_found on | off

使用字段：location,默认值：on

这个参数指定了是否记录客户端的请求出现404错误的日志，通常用于不存在的robots.txt和favicon.ico文件。例如：

  • include： include file | *

你可以包含一些其他的配置文件来完成你想要的功能。例如：include vhosts/*.conf

  • lock_file：lock_file /var/log/lock_file

nginx使用链接互斥锁进行顺序的accept()系统调用，如果Nginx在i386,amd64,sparc64,与ppc64环境下使用gcc,Intel C++,或SunPro C++进行编译，Nginx将采用异步互斥进行访问控制，在其他情况下锁文件会被使用。

  • master_process：master_process on | off

生产环境中不要使用”daemon”和”master_process”指令，这些选项仅用于开发调试。默认值：on。

  • pid：pid file

指定pid文件，可以使用kill命令来发送相关信号，例如你如果想重新读取配置文件，则可以使用：kill -HUP cat /var/log/nginx.pid

  • ssl_engine：ssl_engine engine

这里可以指定你想使用的OpenSSL引擎，你可以使用这个命令找出哪个是可用的：openssl engine -t。

  • timer_resolution：timer_resolution t

这个参数允许缩短gettimeofday()系统调用的时间，默认情况下gettimeofday()在下列都调用完成后才会被调用：kevent(), epoll, /dev/poll, select(), poll()。
如果你需要一个比较准确的时间来记录$upstream_response_time或者$msec变量，你可能会用到timer_resolution。timer_resolution 100ms;

  • try_files：try_files path1 [ path2] uri

按照顺序检查存在的文件，并且返回找到的第一个文件，斜线指目录：$uri /。如果在没有找到文件的情况下，会启用一个参数为last的内部重定向，这个last参数“必须”被设置用来返回URL，否则会产生一个内部错误。

  • user：user user [group]

如果主进程以root运行，Nginx将会调用setuid()/setgid()来设置用户/组，如果没有指定组，那么将使用与用户名相同的组，默认情况下会使用nobody用户与nobody组。默认值：nobody nobody。

  • worker_cpu_affinity：worker_cpu_affinity cpumask [cpumask…]

仅支持linux系统。这个参数允许将工作进程指定到cpu，他调用sched_setaffinity()函数。如下例子：

  • worker_priority：worker_priority [-] number

这个选项可以用来设置所有工作进程的优先级。

  • worker_processes：worker_processes number

由于以下几点原因，Nginx可能需要运行不止一个进程:

* 使用了SMP（对称多处理技术）。
* 当服务器在磁盘I/O出现瓶颈时为了减少响应时间。
* 当使用select()/poll()限制了每个进程的最大连接数时。

在事件模块这一章中我们将使用worker_processes和worker_connextions来计算理论最大连接数(max_clents)：
max_clients = worker_processes * worker_connections。

  • worker_rlimit_core：worker_rlimit_core size

允许的每个进程核心文件最大值。

  • worker_rlimit_nofile：worker_rlimit_nofile limit

进程能够打开的最多文件描述符数

  • worker_rlimit_sigpending：worker_rlimit_sigpending limit

linux内核2.6.8以后，限制调用的进程中真实用户队列可能使用的信号数量。

  • working_directory：working_directory path

程序的工作目录，一般只用来指定核心文件位置，Nginx仅使用绝对路径，所有在配置文件中的相对路径会转移到 –prefix==PATH

事件模块

控制Nginx处理连接的方式

指令

  • accept_mutex：accept_mutex [ on | off ]

Nginx使用连接互斥锁进行顺序的accept()系统调用。

  • accept_mutex_delay：accept_mutex_delay Nms;

如果一个进程没有互斥锁，它将至少在这个值的时间后被回收，默认是500ms。

  • debug_connextion：debug_connection [ip | CIDR]

这个参数可以指定只记录由每个客户端IP产生的debug信息。
当然你也可以指定多个参数。

  • devpoll_changes

这些参数指定了按照规定方式传递到或者来自内核的事件数，默认devpoll的值为32，其余为512。

  • multi_accept：multi_accept [ on | off ]

milti_accept在Nginx接到一个新连接通知后调用accept()来接受尽量多的连接。

  • rtsig_signo

Nginx在rtsig模式启用后使用两个信号，该指令指定第一个信号编号，第二个信号编号为第一个加1。默认值SIGRTMIN+10 (40)

  • worker_connections

worker_connections和worker_proceses（见主模块）允许你计算理论最大连接数：

最大连接数 = worker_processes * worker_connections

在反向代理环境下：
最大连接数 = worker_processes * worker_connections/4
由于浏览器默认打开2个连接到服务器，nginx使用来自相同地址池的fds（文件描述符）与前后端相连接
